# Python 実装の高速化プラン

現行実装(`src/ot_splitting/`)のプロファイル結果と、高速化の余地・実施計画をまとめる。

## 1. 現状の性能

計測環境: 2 vCPU(Linux)、Python 3.13、numpy 2.5 / scipy 1.18。
'obstacle' ケース(N = P = 50、Q = 100、格子点数 25 万/時刻軸込み変数数 約 100 万)。

- **129 ms/反復**(cProfile、30 反復の平均)
- 既定の `--niter 2000` で **約 4.3 分**

### プロファイル内訳(1 反復あたり、実測)

| 処理 | ms/反復 | 割合 | 内容 |
|---|---:|---:|---|
| `prox_j`(3 次方程式) | 41 | 32% | うち `solve_cubic_largest_root` が 36 ms |
| PPXA 更新代数 | ~30 | 23% | `Staggered.__add__/__sub__/__mul__` と `solve_ppxa` 内の numpy 式 |
| 収束診断 | ~20 | 15% | 毎反復の `div_proj(Xu)` + `interp` ×2 + `bb_energy` |
| `div_proj`(prox 側) | ~10 | 8% | Poisson ソルバ(DCT 往復 ~3 ms)+ 勾配減算 |
| `interp_proj` | 14 | 11% | 軸ごとの密行列積 `B @ x`(B は最大 201×201)ほか |
| その他 | ~14 | 11% | `interp`, 配列コピー等 |

### ボトルネックの性質

1. **`solve_cubic_largest_root` が無駄な計算をしている。**
   全格子点で Cardano 分岐(実根 1 つ)と三角関数分岐(実根 3 つ)の**両方**を計算してから `np.where` で選んでいるが、実測では反復中の全点の **100% が Cardano 分岐**に落ちる(prox の 3 次方程式は c ≤ 0 で、密度が非負の領域では判別式がほぼ常に正)。`arccos`/`cos`/`sqrt` の半分は捨てられている。さらに 1 回の呼び出しで 250 万点規模の一時配列を 10 個以上生成しており、メモリ帯域も浪費している。

2. **PPXA 更新がメモリ帯域律速。**
   `Yu[k] = Yu[k] + mu * (2*Zu_bar - Xu - Zu[k])` の形の式が、演算子オーバーロード経由で 1 行あたり 5〜6 個の一時配列(各成分 ~2 MB)を生成する。演算自体は単純な axpy であり、in-place 化・カーネル融合の典型的な対象。

3. **収束診断が毎反復フルコストで走る。**
   `J_hist` などの記録のためだけに、毎反復 `div_proj`(Poisson ソルバ 1 回)+ `interp` 2 回 + `bb_energy` を実行している。ソルバの更新には一切使われない純粋なロギングで、全体の約 15%。

4. **細かい再計算・データ移動。**
   - `poisson_neumann` が固有値の分母配列(25 万要素)を毎回作り直している。
   - `interp_proj` の `pA @ g`:g は境界 2 行以外ゼロなので、実質 (2n+1)×2 の積で済むものを (2n+1)×(n+2) で計算している。`moveaxis`/`concatenate` によるコピーも多い。
   - すべて float64。反復法の途中経過に倍精度は必須ではない。

## 2. 高速化プラン

3 フェーズに分け、各フェーズで数値的な同等性を検証してから次へ進む(検証方法は §3)。

### フェーズ 1: numpy のまま無駄を削る(依存追加なし)（実装済み）

推定効果: **合計 1.7〜2 倍**(129 → 65〜75 ms/反復)

| # | 施策 | 対象 | 推定削減 |
|---|---|---|---:|
| 1-1 | 収束診断を間引く(`--diag-every 25` のようなオプション化。既定は毎反復のままでも可) | `solver.py` | ~20 ms |
| 1-2 | `solve_cubic_largest_root` を Cardano 分岐のみ先に計算し、`delta <= 0` の点が存在する場合だけ三角関数分岐をマスク付きで計算する。一時配列も式の整理で削減 | `prox.py` | ~20 ms |
| 1-3 | PPXA 更新の in-place 化:`Staggered` に `axpy` 相当(`np.add(..., out=)`, `np.multiply(..., out=)`)を導入し、`Yu/Yv` 更新と `Zu_bar` 集約を一時配列なしで行う | `grid.py`, `solver.py` | ~15 ms |
| 1-4 | Poisson の分母配列を `lru_cache`(形状・格子長をキー)で保持 | `poisson.py` | ~3 ms |
| 1-5 | `interp_proj` の `pA @ g` を境界 2 列に縮約(`pA[:, n:] @ [先頭行; 末尾行]`)。`x` の連結を避けて `B` をブロック別に適用 | `projections.py` | ~4 ms |
| 1-6 | `dctn`/`idctn` に `workers=-1` を渡す | `poisson.py` | ~1 ms |

注: 本環境は 2 vCPU のため `workers` の効果は小さい(実測 1.65 → 1.47 ms)。コア数の多い環境では効く。

### フェーズ 2: JIT(Numba)によるカーネル融合

推定効果: **フェーズ 1 からさらに 1.5〜2 倍**(→ 35〜50 ms/反復)

対象は「単純な要素ごと演算だが numpy では一時配列を避けられない」2 カ所:

| # | 施策 | 内容 |
|---|---|---|
| 2-1 | `prox_j` 全体を `@njit(cache=True, fastmath=True)` の単一ループに融合 | msq 計算 → 3 次方程式(分岐はスカラー if で自然に片側だけ実行)→ clamp → obstacle → m の縮小、を 1 パスで。一時配列ゼロ。推定 41 → 5〜8 ms |
| 2-2 | PPXA 更新(反射更新 + 重み付き平均)を njit カーネルに融合 | 全成分を 1 パスで読み書き。推定 30 → 10 ms 程度 |

留意点:

- **依存の追加**: `numba` を optional dependency(`[project.optional-dependencies] jit = ["numba"]`)とし、未インストールなら numpy 実装へフォールバックする構成にする。
- 初回呼び出しに JIT コンパイル数秒(`cache=True` で 2 回目以降は緩和)。
- `parallel=True` は本環境(2 vCPU)では効果が薄いが、コードには入れておく価値がある。
- DCT・行列積は scipy/BLAS のままで良い(Numba 化の対象外)。
- 代替案: GPU が使える環境なら CuPy(`cupyx.scipy.fft` に DCT あり)でほぼ全反復を GPU 化できる。JAX は DCT-II が標準になく実装コストが高い。本プランでは Numba を第一候補とする。

### フェーズ 3: アルゴリズムの変更

数値挙動が変わるため、単体の同等性検証ではなく「最終成果物(evolution.png と J の到達値)の同等性」で判定する。

| # | 施策 | 推定効果 | リスク・備考 |
|---|---|---|---|
| 3-1 | **coarse-to-fine 初期化**: (25,25,50) の粗格子で解いてから解を補間して full 格子の初期値にする。線形補間初期化より遥かに良い初期値になり、full 格子での反復数を削減(2000 → 数百のオーダーを期待) | 2〜4 倍 | 補間(prolongation)の実装が必要。粗格子では迷路の壁(1 セル幅)が潰れるため、obstacle マスクは max-pooling で保守的に粗視化する |
| 3-2 | **早期停止**: `J` と `\|div\|` の停滞を検出して反復を打ち切る(`--tol` オプション) | ケース依存 | 既定は現行どおり固定反復数とし、opt-in にする |
| 3-3 | **float32 化**: 状態変数を単精度にする(メモリ帯域律速な部分がほぼ半減) | 1.3〜1.6 倍 | obstacle ケースは ε = 0 で密度が 0 に張り付くため、3 次方程式の条件数が悪い点で精度劣化の懸念。診断量だけ float64 で計算する等の工夫が必要。検証必須 |
| 3-4 | **Primal–Dual(Chambolle–Pock)ソルバの移植**: MATLAB 版 `test_bb_pd.m` 相当。1 反復が軽く(prox_J + 線形作用素のみ、Poisson 不要)、対角前処理と組み合わせると反復数でも有利な可能性 | 不確実 | 実装コストが最大。フェーズ 1・2 の効果を見てから判断 |

`interp_proj` の密行列積を帯行列ソルバ(O(n²) → O(n))に置き換える案も検討したが、n ≤ 100 では BLAS の密行列積が十分速く(実測 14 ms)、優先度は低い。

## 3. 検証方法

各施策の適用ごとに以下を確認する。

1. **単体テスト**: `uv run pytest`(既存テストがそのまま回帰テストになる)。
2. **数値同等性**(フェーズ 1・2): 固定シード・`niter=50` で新旧実装の `J_hist` / `constr_hist` / 最終 `V` を比較。純粋なリファクタリングは `atol=1e-12` 程度、`fastmath` を使う Numba カーネルは `rtol=1e-8` 程度を許容。
3. **ベンチマーク**: `niter=100` の実行時間(ms/反復)を計測する小スクリプトを `docs/` か `tests/` 併設で用意し、各フェーズの前後で記録する。
4. **成果物**(フェーズ 3): `out/evolution.png` を再生成し、`img/evolution.png` との目視比較と、J の到達値が現行実装と同水準(相対 1% 以内)であることを確認。

## 4. 期待値まとめ

| 段階 | ms/反復(推定) | `--niter 2000` の実行時間 |
|---|---:|---:|
| 現状 | 129 | 約 4.3 分 |
| フェーズ 1 | 65〜75 | 約 2.2〜2.5 分 |
| フェーズ 2 | 35〜50 | 約 1.2〜1.7 分 |
| フェーズ 3(3-1 で反復数 1/4 になった場合) | — | **20〜30 秒** |

フェーズ 1 と 2 は挙動を変えない安全な最適化で確実に効く。最大の伸びしろは反復数そのものを減らすフェーズ 3-1(coarse-to-fine)にある。
